# Launch plan

## The honest version first

You cannot make a repo go viral on demand. What you can do is make a *specific, real* pain
findable at the moment the largest number of people are feeling it — and that part is very
much in your favor right now:

- Cloudflare's June 2026 Bot Management update made AI-agent verification a first-class
  signal, so a lot of agents started getting blocked recently.
- An IETF working group was chartered for Web Bot Auth this year.
- Cloudflare's own implementation is TypeScript and Rust. Most agent code is Python.
- The one Python package named `web-bot-auth` on PyPI (from Fewsats) is a single 0.0.1
  release with an autogenerated placeholder README. `http-message-signatures` (pyauth) is
  solid, but it is generic RFC 9421 with no CLI, no key directory, no diagnostics.

So there is a real hole, at a moment when people are searching. That is as good a setup as
a launch gets. The tool still has to survive contact with the front page.

## Fix these before you post anything

Traffic arrives once. These are cheap and they change conversion:

1. **Set the repo homepage URL.** It is currently empty. Point it at the dev.to post or at
   `https://fastinfer.org`. Free credibility in the sidebar.
2. **Reconsider one word in the README.** It says setting Web Bot Auth up by hand is
   "fiddly" — a word a lot of readers, especially non-native English speakers, will not
   know. "Setting it up by hand takes a surprising amount of trial and error" says the
   same thing to everyone.
3. **Put the 403→200 GIF above the fold** — it already is, and it is the single best asset
   you have. Make sure it also goes on the dev.to post as `cover_image` and on X.
4. **Have 3–5 good first issues open** before launch. People who like a project want
   somewhere to put that energy in the next ten minutes, and an empty issue tracker wastes
   every one of them.
5. **Add a one-line answer to "does this get me past Cloudflare?"** high in the README.
   It is the first question every reader will have, and right now they have to reach the
   Status section to find it.

## The single biggest lever

Everything above is polish. This is the thing that would actually change the ceiling:

**Get one key registered with one real verifier, and show a real 403→200 against a live
CDN-protected site.**

Right now the honest claim is "your side is provably correct." The claim that spreads on
its own is "here is a real blocked request becoming a real allowed one, against
Cloudflare, with a timestamp." That is a screenshot people forward to their team.

Per your notes, the Cloudflare and DataDome forms have not been submitted yet. That is a
manual review with a waiting period, so it is on the critical path — **submit them today,
before you write another line of launch copy.** If the registration lands before the
launch, the post gets significantly stronger. If it lands after, it is your best follow-up
post, and follow-ups to a launch that went well do very well.

## Sequencing

Do not fire everything at once. Each channel feeds the next, and a launch that fails on
HN can be re-run later only if you have not already spent the material.

**Week 0 — preparation**
- Submit the Cloudflare + DataDome registration forms.
- Do the five README/repo fixes above.
- Open the good-first-issues.

**Week 1 — publish the written piece**
- dev.to post goes live (`published: true`). This is your canonical URL and it keeps
  earning search traffic long after the launch spike.
- Post it to r/Python. Low risk, friendly audience, and it shakes out any embarrassing
  bug before the big one.

**Week 2 — Show HN**
- Tuesday–Thursday, 8–10am ET. Submit the repo URL. Post your first comment immediately.
- Then stay at your desk for four hours and answer every single comment. This matters more
  than the title. A submission where the author replies thoughtfully to early skeptics
  routinely outperforms one with a better hook and an absent author.
- Do not ask anyone to upvote. It is detectable and it gets the post buried.

**Week 3 — the long tail, which is where the durable stars come from**
- r/LocalLLaMA and r/AI_Agents with the pain-first framing.
- Answer live questions in the Cloudflare community forum and in GitHub issues where
  people are hitting signed-agent 403s.
- Open a discussion on `cloudflare/web-bot-auth` about the Python implementation.
- Send an implementation report to the IETF working group list.
- PR a short note into one or two agent frameworks' troubleshooting docs.

**Whenever registration comes through**
- The "it works against a real CDN" follow-up post and screenshot. Repost to HN as a new
  submission (this is allowed and normal when there is genuine new substance).

## What success actually looks like

Set the expectation honestly so you can tell signal from noise:

- A Show HN that does *fine* gets 30–80 points and a few hundred stars over a week.
- A Show HN that hits the front page properly gets 200+ and a few thousand.
- Most Show HNs get under 10 points, and that is not a verdict on the code. It is largely
  timing and luck. If it happens, wait a month, change the framing, and resubmit — that is
  explicitly allowed.

The metric that actually predicts whether this project lives is none of those. It is
whether anyone opens an issue that starts with "doctor told me X but the real problem was
Y." That is a person who used it on a real problem, and ten of those are worth more than a
thousand stars.

## One structural idea worth more than any post

The highest-leverage version of this project is not a CLI. It is **a hosted page where
someone pastes a URL and sees the doctor checklist run**, with no install at all.

`doctor` is already the differentiator, it is already a checklist, and its output is
already visual. A public "why is my agent blocked?" checker would be linked from every
forum thread on this topic for years, and every result page is a natural place to say
"install wingfoot to run this against your own key." That is the kind of thing that keeps
earning attention after the launch week ends.

Consider it the next milestone rather than part of this launch.
